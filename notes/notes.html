Basic Media Query Intro

  • Media queries, provide easy logic for applying different styles depending on device characteristics, like its width, height, or device pixel ratio

Adding a Basic Media Query

  • To add a media query link a new stylesheet and set the media attribute
    • <link rel="stylesheet" media="screen and (min-width:500px)" href="over500.css">
    • Stick with either screen or print for the media attribute value

Adding a Basic Media Query 2

  • There are two other ways you can apply media queries:
    • Embed them with an @media tag, or
    • Embed them with an @import tag
  • For performance reasons, you definitely want to avoid @import
  • Linked CSS have many small http requests while @media have few big http requests

Next Step Media Queries

  • Use either min-width or max-width for media queries
  • Avoid using min-device-width and max-device-width because it uses the device's width instead of the browser's width

> Common Responsive Patterns

Intro to Patterns

  • Responsive patterns are constantly evolving but there are some established patterns that are used
  • Four commonly used patterns are:
    1. Mostly Fluid
    2. Column Drop
    3. Layout Shifter
    4. Off Canvas

Pattern - Column Drop

  • Column drop is probably the easiest
  • At its narrowest viewport each element simply stacks vertically
  • As things get wider, the elements expand, at the first breakpoint two elements are now side by side with the third element is underneath
  • At the second breakpoint, three elements are now side by side and as the viewport gets wider margins are now added instead

Pattern - Mostly Fluid

  • The mostly fluid pattern is very similar to column drop, but it tends to be a bit more grid like
  • It has more columns and with columns fitting in different ways, depending on the viewport width

Quiz: Mostly Fluid Part 1

  • Flexbox is such a powerful tool for responsive layout, keep playing with it!

Quiz: Mostly Fluid Part 2

  • Mastering flexbox and grid is crucial to responsive web design

Quiz: Combining Fluid Layouts

  • See pattern-mostly-fluid-blankcss.html file in the supporting materials folder

Pattern - Layout Shifter

  • The layout shifter pattern is probably the most responsive pattern with multiple break points across several different screen widths, but the key to this layout is the way that content moves about instead of reflowing and dropping below other columns
  • Remember, the default order value for an element is zero

Quiz: Which is Which?

  • order and width are super useful, keep using them!

Pattern - Off Canvas

  • With off canvas, instead of stacking content vertically, the off canvas places less frequently used content, for example navigation or app menus, off screen, only showing if the screen is large enough

Responsive Tables Intro

  • Hidden columns, no more tables, and contained tables are three ways to create responsive tables
  • Experiment to see which way works best for your content

Responsive Tables - Hidden Columns

  • Hidden columns essentially hides columns based on their importance as the viewport size gets smaller
  • The biggest problem of hidden columns is that you're hiding content from the user, use this technique with caution
  • If possible, use abbreviated data instead of hiding it completely

Quiz: Hide Some Columns

  • Difference between display: none and visibility: hidden is that using the CSS property/value pair of visibility: hidden hides an element from the browser, but that hidden element still takes up space in the layout
  • However, display: none, removes the element completely from the document and so does not take up any space, even though the HTML for it is still in the source code
  • Read more about the differences here

Responsive Tables - No More Tables

  • With the no more tables technique, below a certain viewport width, the table is collapsed and resembles a long list, as opposed to a table data
  • The nice thing about this technique is that all the data is visible no matter what the size of the viewport is
  • Implement the technique follow the following steps
    1. Set table, thead, tbody, th, td, tr to display as block
    2. Get rid of the thead tr (table header) by using an absolute position off the screen (e.g., top: -9999px; left: -9999px;)
    3. Make room for header by using relative position and by adding padding-left for the td
    4. To add row labels use the before pseudo selector and add td:before using position absolute and content: attr(data-th); to pull the values from the data-th of each of the td elements
  • You can play with this specific example here
  • Learn more responsive table techniques

Responsive Tables - Contained Scrolling

  • One of the easiest things you can do, to contain the table in the viewport is to wrap it in a div and set the width: 100%; and overflow-x: auto;
  • Instead of breaking out of the viewport, the table will instead, take up the same width, but will scroll within in the width of the viewport
  • You can play with this example here

Fonts

  • Use measures as a factor for picking breakpoints
  • Measure is the technical term for the length of a line of text
  • With only a few words per line, it's kind of awkward to read
  • However, if a line of text is too long, readers get lost when trying to find the next line and end up reading the same thing over and over again or might get lazy and start skimming
  • Ideal measure is 45-90 characters per line (cpl) depending on the font used, if it is printed, projected, or on a computer screen
  • But, there seems to be a pretty solid consensus around 65 cpl
  • Font size is alo important to consider, at a minimum use font-size: 16px; and line-height: 1.2em;
  • Depending on the situation, for text heavy sites use font-size: 18px; and line-height: 1.25em;

Minor Breakpoints

  • It's also helpful to add minor breakpoints to make some small changes between major breakpoints
  • For example, it may be helpful to adjust margins and paddings on an element or to increase the font size to make things, a bit easier to read and feel more natural in the layout
  • Use minor breakpoints, to adjust the little things and make things shine!

Introduction to READMEs

  • README is a single file that is used to document smaller projects
  • Take note of the following when reading READMEs:
    • What information is being communicated?
    • How is this information structured?
    • Do you notice any patterns?

Anatomy of a README

  • A README should provide just enough context to get a user up and running with your code
  • Start with a title and description, include logo if feeling extra fancy
  • Next, include any information that is absolutely necessary for understanding your code
  • Remember, there is no particular structure for a README!
  • Few questions to ask yourself when writing a README:
    1. What steps need to be taken?
    2. What should the user already have installed or configured?
    3. What might they have a hard time understanding right away?
  • Often developers will have an installation section
  • Remember you're already familiar with your code but users will not be, so see it from their perspective

Quiz: Documenting a Growing Codebase

  • Lets go over a few common sections and their use cases in a README
  • Including copyright or licensing information, or at least a link to it, is a good idea regardless of project size
  • Contributing section is also a good idea for users that want to help
  • Some services, like ones that check to see if all your tests are passing, will provide special graphics known as shields, for your READMEs
  • These shields are useful because they're provided by a third party, and let your users know helpful information about your code
  • Allow your documentation to grow with the code base and only add new information as needed
  • As a codebase grows, you may want to add additional sections such as: Known Bugs, FAQ, and Table of Contents
  • You can use Choose a License to help you, well... choose a license for your projects

Readable READMEs with Markdown

  • A README file can technically be written in any file format: web page, simple text file, or even a made up format if you wanted to
  • However, generally most use a language called Markdown

Basic Markdown Syntax

  • Markdown is a light markup language often used for READMEs, it is fairly straightforward, and much of the syntax is intuitive
  • But as it turns out, there are may different dialects of Markdown, just like a spoken language, each of these dialects is known as a flavor of Markdown
  • Most of these dialects are pretty much the same, with only minor differences
  • We will be using GitHub Flavored Markdown
  • To make text bold surround it with double asterisks, Isn't today a **wonderful** day?
    • This reads a bit more nicely than a <strong> tag would in HTML, and takes considerably few keystrokes to type out
  • To italicize text, surround it with underscores, And in that moment I thought to myself: _Did I turn off the stove?_
  • Inline code is useful for indicating that you're writing code and not a regular word
    • For this surround text in backticks (`, not a single quote), You should use the `strong` tag.
  • For h1 through h6 tags, all you'll need is a # before your text
    • The number of #s you include tells the Markdown which header tag you're using, ## This is an h2.
  • Basic Markdown documentation
  • GitHub flavored Markdown here

Quiz: Basic Markdown Syntax Quiz

  • A single * may sometimes also work for italicizing text in some flavors of Markdown
  • Make sure you note which flavor of Markdown you're using

More Markdown Syntax

  • Something to keep in mind when using Markdown is that HTML is still valid in Markdown
  • If there's ever something fancy you can't accomplish with just Markdown, it's okay to fall back to HTML, but the catch here is that you may be overcomplicating things
  • Much like how your HTML files should be saved with a .html extension, your Markdown files should be saved with a .md extension
  • Markdown itself can't be opened in the browser like an HTML document, so if you want to preview your Markdown files, Dillinger is a great online resource for you to do so

Document Everything

  • If and when your code grows, you may want to explore other forms of documentation but README is a great place to start
  • Writing great code isn't just about building something that works, but building something that's both easy to use and easy to maintain
  • Make documentation a priority for all your projects moving forward!

Iteration

  • New ways to iterate over an object in ES6 such as the for of loop

Family of For Loops

  • The for loop is obviously the cost common type of loop there is
    • Biggest downside of a for loop is having to keep track of the counter and exit condition
    • While for loops certainly have an advantage when looping through arrays, some data is not structured like an array, so a for loop isn't always an option
  • The for in loop improves upon the weaknesses of the for loop by eliminating the counting logic and exit condition
    • However, you still have to deal with the issue of using an index to access the values of the array, and that stinks; it almost makes itmore confusing than before
    • The for in loop can get you into big trouble when you need to add an extra method to an array (or another object) because for in loops loop over all enumerable properties, this means if you add any additional properties to the array's prototype, then those properties will also appear in the loop
  • The forEach loop is another type of for loop in JavaScript, but forEach() is actually an array method, so it can only be used exclusively with arrays
    • There is also no way to stop or break a forEach loop, if you need that type of behavior in your loop, you'll have to use a basic for loop

For...of Loop

  • The for of loop is used to loop over any type of data that is iterable
  • It's good practice to use plural names for objects that are collections of values
  • That way, when you loop over the collection, you can use the singular version of the name when referencing individual values in the collection
  • You can stop or break a for of loop at anytime using conditionals and you don't have to worry about adding new properties to objects
  • The for of loop will only loop over the values in the object!

> The Document Object Model

Introduction

  • Learn to use JavaScript and the web API called the DOM to control websites

The DOM

  • HTML is received, HTML tags are converted to tokens, tokens are converted to Nodes, Nodes are converted to the DOM
  • The DOM is the full parse representation of the HTML markup
  • The DOM stands for "Document Object Model" and is a tree-like structure that is a representation of the HTML document, the relationship between elements, and contains the content and properties of the elements
  • The DOM is not part of the JavaScript language
  • The DOM is constructed from the browser and is globally accessible by JavaScript code using the document object

Selecting Page Elements With CSS Selectors

  • Being able to select HTML elements is a vital skill when learning ow to access page elements using JavaScript and the DOM!

Select Page Element By ID

  • document.getElementById('footer')
    • Notice, it is 'footer', not '#footer' because it already knows that it's searching for an ID
    • It is called on the document object
    • It returns a single item
    • Passing an unknown ID returns null

Select Page Elements By Class Or Tag

  • document.getElementsByClassName(); and document.getElementsByTagName('span');
    • Both methods use the document object
    • Both returns multiple items
    • The list that's returned is not an array

Notes, Elements, and Interfaces...Oh My!

  • Node (with a capital "N") is a blueprint that contains information about all of the properties and methods for real nodes (with a lowercase "n")
  • If you're not used to the words interface, property, and method, it can be kind of cryptic at first, just remember:
    • interface = blueprint
    • properties = data
    • methods = functionality
  • So the Node Interface is a blueprint for all of the properties (data) and methods (functionality) that every real node has after it's been created
  • Just like the Node Interface, the Element Interface is a blueprint for creating elements
    • The Element Inteface is a descendent of the Node Interface
    • This indicates that the Element Interface inherits all of the Node Interface's properties and methods
  • The .outerHTML and .id properties come from the Element Interface while the .textContent property comes from the Node Interface
  • Node Interface on MDN
  • Element Interface on MDN
  • To check out all of the different interfaces, check here: Web API Interfaces

More Ways To Access Elements

  • Since every browser tries to support the official standards now, jQuery's uses gradually diminished and been replaced by native DOM methods
  • However, out of jQuery's popularity, some new DOM methods were created
  • .querySelector() method tries to select elements just like we do with CSS
    • This method only returns one element!
    • More on the method
  • .querySelectorAll() method returns a list of all elements with a certain class or all of one type of element
    • The list that is returned is a NodeList, it is possible to loop over a NodeList with either its .forEach() method, or the for loop
    • More on the method

> Creating Content with JavaScript

Introduction

  • Start to create, delete, and manipulate JS and the DOM elements

Update Existing Page Content

  • Every element has an .innerHTML property
    • Use this property to get an element's (and all of its descendants) HTML content and to set an element's HTML content
    • It returns a DOMString
  • .outerHTML represents the HTML element itself, as well as its children
  • If you want the text content, use the .textContentproperty
    • The property will set the text content of an element and all its descendants and return the text content of an element and all its descendants
    • Passing any text that looks like HTML to the property will still be displayed as text, it will not be displayed as HTML when the element is rendered
  • .innerText will get the visible text of the element
  • Between .textContent and .innerText, you probably want to use .textContent since that will return all of the text no matter what

Add New Page Content

  • .createElement() to create new elements
  • .appendChild() to add a child element to a parent element as its last child
    • If an element already exists in the DOM and this element is passed to .appendChild(), the .appendChild() method will move it rather than duplicating it
  • .createTextNode() to create a text node
    • An element's .textContent property is used more often than creating a text node with the .createTextNode() method
  • .insertAdjacentHTML() to put HTML text anywhere around an element
    • The method's second argument has to be text, you can not pass an element
    • insertAdjacentHTML on MDN

Remove Page Content

  • .removeChild() must be called on the parent of the element being removed and must be passed the child to be removed
  • .remove() can be called directly on the element to delete
  • The difference between .firstChild and .firstElementChild, is that .firstElementChild will always return the first element, while .firstChild might return whitespace (if there is any) to preserve the formatting of the underlying HTML source code

Style Page Content

  • Least to most specific: rules in a stylesheet -> rules in a <styles> -> rules in a tag's style attribute
  • The web industry as a whole tries to stress the separation of concerns, styles should be in a style sheet, so instead of having styling in your JS file have JS control the class attribute for an element
  • .style.<prop> is used to modify individual styles
  • .style.cssText updates multiple styles at once
    • Write the CSS styles just as you would in a stylesheet
  • .setAttribute() method can be used instead of either .style.<prop> or .style.cssText
    • However, this method is not for just styling page elements (e.g., ID of an element)
  • .className gets/sets a list of classes
    • This property returns a string of all of the element's classes
    • The string can be converted into an array using the .split() method
    • As an array you can use for loops, the .push() method to add an item to the list, or .pop() to remove an item from the list
  • .classList gets/sets/toggles CSS classes
    • New than the .className property, but much easier to use
    • Returns a DOMTokenList
    • Has number of properties of its own: .add(), .remove(), .toggle(), .contains()
  • Article: CSS Specificity: Things You Should Know

> Working with Browser Events

Introduction

  • Learn about what events are, how to respond to events, what data is stored in an event, and how to stop an event

Respond to Events

  • The Node Interface inherits from the EventTarget interface
  • EventTarget is an interface implemented by objects that can receive events and may have listeners for them
  • Elements, document, and window are the most common event targets, but other objects can be event targets too
  • The EventTarget Interface, doesn't have any properties and only three methods! These mehods are:
    • .addEventListener()
    • .removeEventListener()
    • .dispatchEvent()
  • An event listener needs three things:
    1. an event target - this is called the target
    2. the type of event to listen for - this is called the type
    3. a function to run when the event occurs - this is called the listener
  • monitorEvents() function is useful to use when working with events
